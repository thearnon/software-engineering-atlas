## RBAC คืออะไร

RBAC หรือ Role-Based Access Control คือวิธีจัดสิทธิ์โดยให้ผู้ใช้ได้รับ role แล้ว role นั้นกำหนดว่า action ไหนทำได้ ทำไม่ได้ หรือทำได้แบบมีเงื่อนไข

แกนของ RBAC คือไม่ผูก permission เข้ากับ user ทีละคน แต่ผูก permission เข้ากับ role แล้วค่อยให้ user ได้ role ตามหน้าที่ในระบบ

คำถามหลักของ RBAC คือ "ในบริบทนี้ user มี role อะไร และ role นั้นทำ action นี้ได้หรือไม่"

<Callout type="definition" title="จำสั้น ๆ">
  RBAC แยก user, role และ permission ออกจากกัน ทำให้ระบบจัดสิทธิ์ได้เป็นกลุ่ม
  review ได้ง่าย และลด logic กระจัดกระจายตามหน้าจอ
</Callout>

## ทำไม RBAC สำคัญในระบบองค์กร

ใน enterprise internal web application สิทธิ์ไม่ได้มีแค่ซ่อนปุ่ม แต่กระทบข้อมูลจริง workflow จริง และความรับผิดชอบของคนในองค์กร

ระบบแบบ approval, dashboard, KPI report, admin console และ audit review ต้องตอบให้ได้ว่าใครดูข้อมูลได้ ใครอนุมัติได้ และใครแก้ configuration ได้

ถ้าไม่มี RBAC ที่ชัด ระบบมักจบด้วยการ hard-code เงื่อนไขไว้หลายจุด พอ role เปลี่ยนหรือ business rule เปลี่ยนก็แก้ยากและ test ยาก

<Callout type="enterprise" title="บริบท enterprise">
  RBAC มักต้องออกแบบพร้อม approval workflow, permission matrix และ audit log
  เพราะสิทธิ์ของ action หนึ่งมักขึ้นกับทั้ง role, state และ scope ของข้อมูล
</Callout>

## Case ตัวอย่าง: ระบบขอเปลี่ยน credit limit

สมมติองค์กรมีระบบให้พนักงานส่งคำขอเปลี่ยน credit limit ของลูกค้า คำขอหนึ่งรายการมี status เช่น draft, submitted, approved, rejected และ returned

Requester สร้างคำขอ แก้ draft และ submit ได้ แต่หลัง submit แล้วไม่ควรแก้เองได้ เว้นแต่ Approver ส่งกลับมาให้แก้

Approver อนุมัติหรือ reject ได้เฉพาะ request ที่อยู่ใน scope ของตัวเอง เช่น แผนกเดียวกัน วงเงินไม่เกิน limit และไม่ใช่ request ที่ตัวเองสร้าง

Admin ดูแล role และ configuration ได้ แต่ไม่ควรถูกใช้เป็น role สำหรับข้ามทุก business rule เพราะจะทำให้ audit และ security review อธิบายยาก

Auditor อ่าน request และ audit log ได้เพื่อสอบกลับ แต่ไม่ควรเปลี่ยนข้อมูลธุรกิจหรืออนุมัติแทนคนใน workflow

### สิ่งที่ case นี้สอน

- Role บอกว่าใครควรทำ action ได้
- Status บอกว่า action นั้นทำได้ในช่วงไหนของ workflow
- Scope บอกว่า role นี้ทำได้กับข้อมูลชุดไหน
- Policy บอกข้อห้ามที่ role อย่างเดียวตอบไม่ได้
- Audit log บอกว่า action สำคัญเกิดขึ้นจริงโดยใครและเมื่อไร

## จาก case สู่ Permission Matrix

เมื่อแปลง case นี้เป็น permission matrix เราจะได้ตารางที่มี role เป็นแถว action เป็นคอลัมน์ และ cell เป็นผลลัพธ์เช่น Allow, Limited หรือ Deny

ตัวอย่างเช่น Approver อาจได้ Allow สำหรับ Approve within scope แต่ Requester ต้องเป็น Deny สำหรับ action เดียวกัน

ส่วน Admin อาจได้ Allow สำหรับ Manage role config แต่ไม่ได้แปลว่าควร Allow สำหรับ Approve within scope เพราะ admin permission กับ business approval เป็นคนละเรื่อง

Permission matrix ด้านล่างจึงไม่ใช่แค่ตารางสิทธิ์ แต่เป็นเครื่องมือคุยร่วมกันระหว่าง business, developer, QA และ security review

## Common Mistakes

- เริ่มจาก job title ในองค์กรโดยตรง ทำให้ role เยอะและซ้ำกันเกินไป
- เช็ก permission เฉพาะใน UI แต่ backend หรือ API ยังเรียกได้อยู่
- ให้ Admin ทำได้ทุกอย่างจนแยก business action กับ system configuration ไม่ออก
- ลืม scope เช่น branch, department, owner หรือ amount limit
- ไม่เก็บ denied action สำคัญ ทำให้สอบกลับไม่ได้ว่ามีคนพยายามทำอะไรแต่ถูกปฏิเสธ
- ไม่ผูก RBAC กับ test case ทำให้ permission regression หลุดง่ายตอนแก้ระบบ

## Checklist ตอนออกแบบ RBAC

ใช้ checklist นี้ตอนเริ่มออกแบบหรือ review ระบบสิทธิ์

- ระบุ action สำคัญเป็นคำกริยา เช่น create, submit, approve, reject, configure, view log
- แยก role จาก job title ให้เหลือ role ที่มีความหมายในระบบจริง
- ระบุ scope ของแต่ละ role เช่น own record, department, branch, amount limit
- แยก system admin ออกจาก business approver
- นิยาม Allow, Limited และ Deny ให้ทีมเข้าใจตรงกัน
- เขียน permission matrix ก่อนลง code สำหรับ workflow สำคัญ
- สร้าง test case จาก matrix ทั้งกรณี allow, deny และ limited
- ระบุ action ที่ต้องเข้า audit log ตั้งแต่ช่วง design

<Callout type="tip" title="วิธีคิดที่ใช้ได้จริง">
  เริ่มจาก action สำคัญใน workflow ก่อน แล้วค่อยถามว่า role ไหนควรทำได้
  อย่าเริ่มจากการสร้าง role จำนวนมากตั้งแต่แรก
</Callout>

## อ่านต่อจากหัวข้อนี้

ถ้าต้องการต่อยอดจาก RBAC ให้อ่านตามลำดับนี้

- Permission Matrix: แปลง role และ action ให้เป็นตารางที่ review และ test ได้
- Approval Workflow: ดูว่า action แต่ละตัวเกิดใน state ไหนของคำขอ
- Audit Log: ออกแบบหลักฐานย้อนหลังว่าใครทำอะไร เมื่อไร และในบริบทไหน

## Viewer ที่เกี่ยวข้อง
