## How it appears in enterprise systems

A permission matrix shows which roles can perform which actions in a system.

In enterprise products, the matrix gives product, business, development, QA, and security reviewers one shared view of access rules.

## How it connects to RBAC

RBAC is the access-control model. A permission matrix makes that model visible, reviewable, and testable.

A useful matrix connects to workflow states, audit logs, and test cases instead of living as a separate document nobody maintains.

## Things to watch

- Do not use the matrix as a substitute for all domain rules.
- Name actions clearly, such as view, create, approve, reject, and configure.
- Distinguish allow, limited, and deny states.
- Use the matrix as input for testing and release review.
